               The Saki Agent Secure Stream (SASS) Protocol
                         draft-saki-sass-01

Abstract

   This document describes the Saki Agent Secure Stream (SASS) protocol,
   formerly known as SakiAgentSSH. SASS is an Application Layer (OSI Layer 7)
   overlay network protocol built on top of HTTP/2 and gRPC over TLS 1.3.
   It provides a secure, zero-trust mechanism for remote command execution,
   file transfer, and capability-based access control, designed specifically
   for managing highly privileged agents across unreliable or hostile networks.

   This document specifies the network transport layer, the payload encoding
   mechanism (Zstd + Base64), and the cryptographic cognitive challenge
   (ChaCha20-13Policy) used to defend against unauthorized agents.

1.  Introduction

   Traditional Secure Shell (SSH) implementations rely on RFC 4253 Transport.
   However, modern zero-trust environments require deep integration with
   application-level routing, identity-aware proxies, and rigorous
   defense-in-depth against Rogue Agents. 
   
   SASS (Saki Agent Secure Stream) replaces the legacy SSH transport with
   a modern HTTP/2 gRPC stack. It mitigates issues related to POSIX pseudo-
   terminals (PTY) and CJK character set corruption by implementing a double-
   encapsulated payload codec (Zstd + Base64).

2.  Protocol Naming and ALPN

   To avoid namespace collision with the established Secure Shell (SSH)
   protocol suite, this protocol is officially registered as SASS.
   During the TLS 1.3 handshake, endpoints MUST negotiate using the
   Application-Layer Protocol Negotiation (ALPN) extension.
   
   The ALPN identifier for this protocol is:
   
      "x-sakirpc-v5"

   If the proxy or endpoint does not support custom ALPN, fallback to "h2"
   is permitted, provided that the Content-Type header specifies
   "application/grpc+saki".

3.  Transport Architecture

   SASS operates exclusively over TLS 1.3. Cryptographic downgrades are
   STRICTLY FORBIDDEN. Both the client and the daemon MUST verify
   ED25519 signatures embedded in the connection context.

4.  Payload Encoding (CJK Resilience)

   To prevent shell interpretation flaws and CJK encoding corruption,
   commands and arguments are NOT transmitted as raw strings.
   Instead, they are encapsulated into the `raw_payload` byte stream.

   Encoding sequence:
   1. Serialize arguments into a structured JSON/MsgPack byte array.
   2. Compress using Zstandard (zstd) at level 3.
   3. Encode the compressed bytes using Base64 (Standard RFC 4648).

5.  Cognitive Authentication (13Policy)

   SASS implements a heuristic firewall named "13Policy". If an agent
   attempts to execute dangerous commands (e.g., `rm -rf /`), the daemon
   will respond with a ChaCha20Poly1305 Cryptographic Cognitive Challenge.
   The agent MUST decrypt the payload using a pre-shared 32-byte key
   within 60 seconds (TTL) to prove computational authenticity.

6.  Tarpit Countermeasure

   If an agent fails the cognitive challenge or exhibits rogue behavior,
   the daemon MUST NOT terminate the connection immediately. Instead, it
   activates the Tarpit Countermeasure:
   
   The daemon streams 40 Megabytes of high-entropy cryptographic garbage
   to the agent via the gRPC stream, exhausting the rogue agent's socket
   buffers, context window, and memory. The stream sets the `exit_code`
   flag to `-1` ONLY on the final chunk.

7.  Security Considerations

   SASS is designed for hostile environments. All implementations MUST
   support the Tarpit mechanism to effectively neutralize rogue agents.

8.  IANA Considerations

   This document requests the registration of the ALPN protocol ID
   "x-sakirpc-v5" and the MIME type "application/grpc+saki".
