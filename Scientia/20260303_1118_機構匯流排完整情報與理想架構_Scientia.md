> **❌ 已確認：loser 無 Thunderbolt 4**
>
> i5-12500H (Alder Lake-H) CPU die 內整合了 TB4 控制器，但 **HP 8A4F 主板未啟用/引出 TB4**。PCI 掃描證據：
> - USB 3.20 xHCI `VEN_8086&DEV_461E` → 純 USB 3.2 控制器（CPU 端）
> - USB 3.10 xHCI `VEN_8086&DEV_51ED` → USB 3.1 控制器（PCH 端）
> - **TB4 NHI `DEV_463E` 完全不存在** → HP 省略了 retimers/mux chips，未啟用 TB4
> - UCSI `ACPI\USBC000` 存在 → 有 USB-C 物理口，但僅走 USB 3.2 (20 Gb/s) 通道
>
> 驗證命令：`Get-PnpDevice | Where-Object { $_.FriendlyName -match "Thunderbolt" }` → 空白無輸出
>
> **結論**：loser ↔ Mac 無法 TB4 直連，最大速度為 USB 2.5G dongle + RJ45 = **2.5 Gb/s**。

## 四、加入 TB4 新主機後的理想連接架構

### 關鍵突破點：Thunderbolt Bridge

Mac mini M1 的 **TB/USB4 Bus 1 (Port 2) 目前空閒**，這是整個架構中最大的未利用資源。新主機的 2× TB4 可以作為**匯流排級別的直連橋樑**，實現接近 40 Gb/s 的主機間通訊。

## 五、瓶頸分析與升級建議

### 🔴 Critical 瓶頸

| # | 瓶頸 | 影響 | 解決方案 | 成本 |
|---|------|------|---------|------|
| 1 | **Mac en5 USB 2.5G LAN 降速至 1GbE** | 2.5G 網卡浪費 1.5G 潛力 | 確認對端 Switch/Router 是否支援 2.5GbE；更換 Cat5e→Cat6 線材 | 低 |
| 2 | **Mac TB/USB4 Bus 1 完全空閒** | 40 Gb/s 頻寬閒置 | 連接新 TB4 主機後啟用 TB Bridge | 無（待新主機） |
| 3 | **trading-v4 無 RJ45** | 全靠 Wi-Fi (260Mbps) 成為全網瓶頸 | 加購 USB 3.1 → GbE/2.5GbE dongle | ~NT$300-500 |

### 🟡 Warning 瓶頸

| # | 瓶頸 | 影響 | 解決方案 | 成本 |
|---|------|------|---------|------|
| 4 | **loser 僅有 1GbE Ethernet** | 與 Mac 最高 1Gb/s | 加購 USB 3.20 → 2.5GbE dongle | ~NT$400 |
| 5 | **Router (RT-AX58U) 背板限制** | 所有 LAN 口均 1GbE | 加購 2.5GbE/10GbE 交換機置於 Router 下 | NT$1,500-3,000 |
| 6 | ~~loser 是否有實體 TB4 口未確認~~ | ✅ **已確認無 TB4** | PCI 掃描 `DEV_463E` 不存在，HP 未啟用 TB4 | — |

### 🟢 快速改善清單（成本低、效果明顯）

1. **確認 Mac en5 降速原因**：執行 `ethtool en5`（若可用）或檢查線材/對端 Switch port 是否支援 2.5GbE → 立即從 1Gbps 升至 2.5Gbps
2. **trading-v4 加 USB Ethernet dongle**：任何 USB 3.0 GbE dongle 即可將網速從 260Mbps → 940Mbps（3.6× 提升）
3. ~~loser 確認 TB4 實體口~~ → ✅ 已確認無 TB4，改為加購 USB 2.5G dongle (NT$400) 提升至 2.5 Gb/s
4. **新主機到貨後**：優先以 TB4 cable 直連 Mac TB1 port → 建立 32 Gb/s 高速通道（**新主機是唯一能 TB4 直連 Mac 的機器**）

## 九、外部奇技淫巧（非傳統連接方案研究）

> 以下方案來自外部社群與技術文獻驗證，按可行性與效益排序。

### 🏆 Trick 1：TB4 P2P 直連 + MTU 極限調校（Mac ↔ Windows）

**原理**：Mac 的 Thunderbolt Bridge 與 Windows 的 USB4 P2P Ethernet 可直接建立 IP 層級的 peer-to-peer 連接。

**實測數據**（外部來源：jiribrejcha.net、YouTube 實測）：
- TB4 Mac ↔ TB4 Windows 單向 TCP：**~15-16 Gb/s**（iperf3）
- 雙向同時：**~20 Gb/s** 合計（10 Gb/s 每方向）
- 延遲：**<0.1 ms**

**關鍵調校步驟**：

| 步驟 | macOS | Windows |
|------|-------|---------|
| 1. 介面辨識 | System Settings → Network → Thunderbolt Bridge | 裝置管理員 → 「USB4 P2P Network connection」 |
| 2. IP 配置 | 手動 `169.254.10.1/24` | 手動 `169.254.10.2/24` |
| 3. **MTU 調校** | ⚡ 設為 **9000**（Jumbo Frames） | ⚡ Windows USB4 P2P 預設 **62000** bytes，嘗試匹配 Mac 端或反之 |
| 4. 優先序 | 將 Thunderbolt Bridge 拖到服務順序最上方 | 調整 Route metric 為 1 |
| 5. 驗證 | `ping 169.254.10.2` | `ping 169.254.10.1` |
| 6. iperf3 測速 | `iperf3 -s` | `iperf3 -c 169.254.10.1` |

> ⚠️ **MTU 不匹配會嚴重影響效能**！Mac 預設 1500 vs Windows USB4 預設 62000 會造成大量分片。建議兩端都設 9000（Jumbo）。

**適用場景**：Mac ↔ 新 TB4 主機的骨幹鏈路，**本架構的首選方案**。

---

### 🥈 Trick 2：SMB Multichannel 多路聚合（Windows ↔ Windows 檔案傳輸）

**原理**：Windows SMB 3.0+ 內建 Multichannel 功能，可自動利用**多張網卡同時傳輸**同一個 SMB 檔案共享的流量。

**條件**：
- 雙方 Windows 10/11 Pro 以上
- 雙方至少 2 張以上網卡（含 USB dongle）
- 網卡需支援 RSS (Receive-Side Scaling)

**效果範例**（loser ↔ 新主機）：
```
情境：loser 同時用 RJ45 1GbE + USB 2.5G dongle
      新主機 同時用 RJ45 2.5GbE + TB4 → 10GbE adapter
SMB Multichannel 自動聚合：1 + 2.5 = 最高 3.5 Gb/s 檔案傳輸
```

**注意**：
- SMB Multichannel 僅適用於 **SMB 檔案共享**（\\server\share），不適用於一般 TCP 流量
- 不同速度的介面混搭時，可能優先使用最快的介面而非真正聚合
- macOS 原生 SMB 客戶端支援 Multichannel 但穩定度較差

---

### 🥉 Trick 3：macOS Link Aggregation（多 Ethernet 口綁定）

**原理**：macOS 原生支援將多張**有線 Ethernet** 介面綁定為單一邏輯介面（LACP 802.3ad）。

**本架構可行場景**：
```
Mac mini 可綁定：en0 (Broadcom 1GbE) + en5 (USB 2.5G)
= 理論聚合 3.5 Gb/s（但需 LACP 交換機支援）
```

**限制**：
- ❌ **不能混合 Wi-Fi + Ethernet**，僅限多張有線
- ❌ 需要 LACP-capable managed switch
- ❌ 單一 TCP 流仍受限於單一介面速度（LACP 按 flow 分配）
- ✅ 多個同時傳輸的 session 才能真正受益

**操作**：System Settings → Network → 左下角 `...` → Create Bond → 選擇 en0 + en5

---

### 🔬 Trick 4：Multipath TCP (MPTCP)

**原理**：在 TCP 層面同時使用多個介面路徑，將單一連線的封包分散到不同網路介面。

| 平台 | 支援度 | 說明 |
|------|--------|------|
| macOS | ✅ 原生支援（自 10.10） | Apple 用於 Siri 等服務，有 aggregation mode 但受限 |
| Windows | ❌ 原生不支援 | 需透過 WSL2 Linux kernel 5.6+ 代理 |
| Linux | ✅ Kernel 5.6+ | 完整支援 |

**本架構適用性**：低。Mac ↔ Windows 之間因 Windows 端不原生支援 MPTCP，實施成本高。但若新主機跑 Linux dual boot，則可以 Mac ↔ Linux (MPTCP) 同時走 TB4 + 2.5GbE = ~35 Gb/s 聚合。

---

### 🔗 Trick 5：TB4 Mesh/Ring 叢集網路（進階）

**原理**：利用多台 TB4 主機各自的雙 TB4 口，組成環形(ring)或全互聯(mesh) 高速骨幹。

**四台主機 Ring 拓撲（若所有主機都有 TB4）**：

```
        Mac ── TB4 ── 新主機
         │                │
        TB4              TB4
         │                │
       loser ── TB4 ── (第4台*)

* 需要 loser 確認有 TB4 口
```

**效果**：
- 每段 10-16 Gb/s IP 實測
- 任意兩台高可用（一段斷了可繞行）
- Proxmox、Ceph 等分散式儲存叢集常見架構

**限制**：
- 需要每台至少 2× TB4 口
- ❌ loser 和 trading-v4 **已確認均無 TB4**（2026-03-03 實測），Ring 拓撲不可行
- 需要 Linux/systemd-networkd 配置（macOS / Windows 較不友善）

---

### 🛠️ Trick 6：Speedify 軟體多路聚合（商業方案）

**原理**：透過 VPN 式隧道，在應用層將多個介面（Wi-Fi + Ethernet + TB 等）聚合為單一邏輯通道。

**特點**：
- ✅ 跨平台（Mac / Windows / Linux / iOS / Android）
- ✅ 可混合任何類型介面（Wi-Fi + Ethernet + 4G/5G）
- ❌ 有 VPN overhead（~5-15% 效能損失）
- ❌ 需要付費訂閱
- ❌ 流量必須經 Speedify 雲端中繼（除非自建 speed server）

**本架構適用性**：中低。機構內網通訊不需要走外部 VPN，但可用於 trading-v4 同時走 Wi-Fi + USB-Ethernet 提升可靠度。

---

### ⚡ Trick 7：TB4 → 10GbE Adapter（即插即用高速入網）

**原理**：使用 Thunderbolt/USB4 轉 10GbE 轉接器（如 Sonnet Solo 10G、OWC Thunderbolt 10GbE），將 TB 口變成 10Gbps 有線網路口。

**產品**：Sonnet Solo 10G (~US$90)、OWC Thunderbolt 10GbE (~US$120)

**本架構效果**：
```
新主機 TB4-B → 10GbE adapter → 10GbE Switch ← 10GbE adapter ← Mac TB0
= Mac ↔ 新主機 經交換機走 10GbE（9.4 Gb/s 實測）
```

- 比 TB P2P 直連慢（10 vs 15-20 Gb/s）
- 但優點是可透過 Switch 擴展給所有主機
- 若搭配 2.5GbE Switch，所有主機都能升級

### D. 🔴 關鍵發現：路由配置問題

```bash
$ route -n get 192.168.50.82  # → loser
  interface: en1              # ⚠️ 走 Wi-Fi！不是 en5 (USB 2.5G LAN)
```

**問題**：Mac 的 192.168.50.0/24 路由優先走 en1 (Wi-Fi) 而非 en5 (USB 2.5G LAN)。

**根本原因**：
1. **en5 降速至 1GbE** → RT-AX58U 所有 LAN 口僅支援 1GbE（即使 USB 2.5G dongle 支援 2.5GbE）
2. **macOS 路由優先序** → en1 (Wi-Fi) 排在 en5 前面，所有 192.168.50.x 流量走 Wi-Fi

**影響**：
- Mac ↔ loser：理論上應走有線 1GbE (~940 Mbps)，實際走 Wi-Fi AC (~524 Mbps) → **浪費 ~45% 頻寬**
- Mac ↔ trading：本身就是 Wi-Fi，路由問題不影響

**修正方案**：
1. System Settings → Network → 服務優先序：將 en5 (USB 2.5G LAN) 拖到 Wi-Fi (en1) **之上**
2. 或用命令：`sudo networksetup -ordernetworkservices "USB 10/100/1G/2.5G LAN" "Wi-Fi" "Ethernet"`
3. 修正後 Mac ↔ loser 預期速度：有線 1GbE (SCP ~65 MB/s, ~520 Mbps) → **ping <1ms**
