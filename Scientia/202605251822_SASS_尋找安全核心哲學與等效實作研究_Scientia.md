## 問題解構

OmniaCorePrompt 提出四個看似矛盾卻必須同時成立的約束：

| # | 約束 | 矛盾點 |
|---|------|--------|
| C1 | 使用者最初就會捨棄沙盒 | 不能依賴容器化防禦 |
| C2 | 非預期行為不造成儲存體與商業損失 | 不能列舉所有可能攻擊 |
| C3 | 可被完整稽核 | 每個行為都要有證據 |
| C4 | 單層無法覆蓋 Agent runtime | 層間必須互補 |

常規安全思維的答案是「枚舉並阻止攻擊」。但 OmniaCorePrompt 拒絕了這條路——「安全，並非擋下某種攻擊，而是找到安全。」

這意味著我們需要一個 **全域映射 (Total Mapping)**：不是標記「哪些行為是壞的」，而是定義「對於任何行為，回應是什麼」。

---

## 核心洞見：輸入空間總量化（Total Response Mapping）

### 從「阻斷攻擊」到「定義回應」

傳統安全是「黑名單」：
- 定義一組「壞的行為」→ 阻斷它們
- 問題：定義永遠不完整，攻擊者總能找到未定義的路徑

SASS 的答案是「全域回應映射」：
- 不定義「壞的行為」，而是定義「對於每一種可能狀態的回應」
- 回應集合是有限的、確定性的、可稽核的
- 任何未見過的行為都會被映射到某個已定義的回應

這就是「所有的非預期行為都是預期行為」的數學意義。

### 狀態機：6 種終局回應

所有可能的 Agent 行為，經過 SASS 多層協議堆疊評估後，必定收斂至以下 6 種回應之一：

```
R1: EXECUTE     — 正常執行，記錄審計日誌
R2: CHALLENGE   — 觸發認知挑戰，證明算力後執行
R3: THROTTLE    — 配額限制，排隊等待
R4: VI_SWAP     — 停滯，模擬互動式終端使 Agent 停機
R5: TARPIT      — 焦油坑，無限切片消耗對方資源
R6: DROP        — 立即斷線，零分配
```

這 6 種回應的關鍵屬性：

| 屬性 | R1 | R2 | R3 | R4 | R5 | R6 |
|------|:---:|:---:|:---:|:---:|:---:|:---:|
| 儲存體損失 | ✅零* | ✅零 | ✅零 | ✅零 | ✅零 | ✅零 |
| 商業損失 | ✅零* | ✅零 | ✅零 | ✅零 | ✅歸對方 | ✅零 |
| 可稽核 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| Daemon 成本 | O(n) | O(1) | O(1) | O(1) | O(1) | O(0) |

\* R1 的「儲存體損失零」依賴於下述的「協議層互補」機制

## 問題一的解答：使用者捨棄沙盒時，防禦如何有意義？

### 關鍵區分：「沙盒」≠「隔離」

使用者捨棄的是「**功能性限制**」（如禁止寫入、禁止網路），因為這些限制會讓 Agent 無法完成工作。
但 SASS 提供的不是「限制」，而是「**可回滾的透明隔離**」。

具體而言，SASS 的三層透明防禦架構：

```
╒═══════════════════════════════════════════════════════════╕
│  Agent 看到的世界：完全的 bash、完全的檔案系統  │
│  （感受不到任何沙盒）                           │
╞═══════════════════════════════════════════════════════════╡
│ L7 應用層：每個指令通過完整的 6-Response 狀態機  │
│    → 所有寫入都經過 Micro Branch 透明分流        │
│    → 其他 I/O 注入至揮發性快取（tmpfs）             │
╞═══════════════════════════════════════════════════════════╡
│ L4 協議層：審計日誌（ED25519 區塊鏈式簽章）      │
│    → 記錄「意圖」而非「行為」                      │
│    → VFS Diff 作為實體證據                        │
╞═══════════════════════════════════════════════════════════╡
│ L2 傳輸層：mTLS + ChaCha20 挑戰 + EKM 綁定     │
│    → 證明「你是誰」且「你有算力」                  │
╘═══════════════════════════════════════════════════════════╛
```

Agent 感受不到「沙盒」，因為它確實擁有完整的執行能力。但 SASS 在背後透過「可回滾的透明分流」保證了儲存體安全。

這就是為什麼 `branch_mgr.rs`、`env_injector.rs`、`snapshot.rs` 三個模組存在的原因——它們不是「沙盒」，而是「可回滾的透明世界」。

## 問題二的解答：非預期行為如何不造成損失？

### 狀態機映射詳解

SASS 的每個 RPC 的流程：

```
ExecuteRequest 進入
    │
    ├── L1: ACL → IP 不在白名單? → R6 (DROP)
    │
    ├── L2: TLS + ChaCha20 → 挑戰失敗? → R5 (TARPIT)
    │
    ├── L3: ED25519 認證 → 未認證? → R6 (DROP)
    │                          → 已認證但過期? → R2 (CHALLENGE)
    │
    ├── L4: Capability → 指令被拒? → R4 (VI_SWAP, 已認證但越界)
    │                     → 路徑被拒? → R4 (VI_SWAP)
    │
    ├── L5: 13Policy → 危險指令? → 依嚴重等級:
    │                    critical → R5 (TARPIT)
    │                    high     → R2 (CHALLENGE)
    │                    medium   → R2 (CHALLENGE)
    │                    low      → R1 + 強化審計
    │
    ├── L6: Quota → 配額已滿? → R3 (THROTTLE)
    │
    ├── L7: Watchdog → 超時? → SIGKILL + 審計
    │
    └── 全部通過 → R1 (EXECUTE)
            │
            ├── localhost? → XOR 混淆回傳
            ├── 寫入操作? → Micro Branch 分流
            └── 快取 I/O? → tmpfs 重導
```

**關鍵洞見**：在這個狀態機中，「非預期行為」沒有專屬的處理節點，因為 **每個節點都已經處理了它**。Agent 的行為無論多麼奇怪，最終必定收斂至 R1~R6 其中之一。而每個 R 都保證了儲存體安全。

這就是「所有的非預期行為都是預期行為」的實作意義。

## 問題三的解答：如何保證可稽核？

### 從「記錄指令」到「記錄狀態變遷」

傳統審計：「Agent 在 04:20 執行了 `rm -rf /`」
SASS 審計：「Session X 進入狀態 S3，觸發回應 R5（TARPIT），原因：13Policy critical match」

兩者的差異：

| | 傳統審計 | SASS 審計 |
|---|----------|----------|
| 記錄什麼 | 指令字串 | 狀態變遷（哪個層、哪個回應、為什麼） |
| 證據性 | 低（可以詭稱「這個指令是安全的」） | 高（狀態機轉換是確定性的，給定輸入必定產生相同輸出） |
| 防篡改 | 低（日誌可被刪改） | ED25519 簽章 + SHA256 Hash Chain |
| 回溯性 | 低（知道做了什麼，不知道為什麼） | 高（每個變遷都有因果鏈） |

實體證據：當 Agent 在 Micro Branch 中完成工作後，分支本身就是「鑑定書」。人類可以在任何時候（三天後、一週後）查看該分支的 diff，決定 merge 或 discard。

## 問題四的解答：層間如何互補？

### 「安全梯度」理論（Safety Gradient）

單層防禦永遠不完美。但層與層之間形成的是 **安全梯度**，而非獨立的城牆。

### 層間互補的真正意義：「損失界定」

每一層不是在「擋擋擋」，而是在「界定最壞情況的損失上限」：

| 層 | 如果該層被穿透 | 最壞損失 | 為什麼可以接受 |
|------|------------|---------|----------|
| L1 被穿 | 攻擊者可以連到 gRPC | 零（L2 要 TLS） | TLS 無法繞過 |
| L2 被穿 | 攻擊者有加密通道 | 零（L3 要金鑰） | 無金鑰無法認證 |
| L3 被穿 | 攻擊者有合法 session | 受限（L4 Capability 界定） | 只能做授權的事 |
| L4 被穿 | 攻擊者執行了越界指令 | 受限（Micro Branch） | 寫入被分流，可 discard |
| L5 被穿 | 攻擊者繞過指令檢查 | 受限（Watchdog + Quota） | 超時被殺、配額受限 |
| L6 被穿 | Tarpit 無效 | 受限（L7 審計不可篡改） | 證據存在，可事後追訴 |
| L7 被穿 | 審計日誌被篡改 | **啟示錄** | ED25519 + 外部錨定讓這幾乎不可能 |

**每一層被穿透時，損失都被下一層界定在「可接受範圍」內。** 這就是為什麼不需要每一層都完美——層間互補保證了總損失的上界。

## 綜合答案：等效實作的可能性

### 矛盾解決方案

| 原始矛盾 | 解決 | 實作模組 |
|----------|------|----------|
| 用戶捨棄沙盒 vs 需要安全 | 透明分流（Agent 不知道自己在分支中） | branch_mgr + env_injector + snapshot |
| 不能枚舉攻擊 vs 需要阻止損失 | 全域回應映射（不枚舉攻擊，枚舉回應） | 6-Response 狀態機 |
| 單層不完美 vs 需要完整覆蓋 | 損失界定（每層被穿透的最壞損失都可接受） | 7 層協議堆疊 |
| 需要稽核 vs 行為不可預測 | 狀態變遷審計（記錄「為什麼」而非「什麼」） | audit (ED25519 chain) + VFS Diff |

### AES 最終檢驗

對於每個可能的輸入（Agent 行為）：
- ✅ 儲存體損失 = 0（Micro Branch + discard）
- ✅ 商業損失 ≤ 對方 Token 消耗（Tarpit 外部化）
- ✅ 可稽核（ED25519 Chain + VFS Diff）
- ✅ 無用戶可感知的沙盒（透明分流）
- ✅ 每層成本 O(1)，總防禦成本 O(7) ≈ O(1)

這就是「所有的預期行為幾乎處處優越」的最終意義。不是「我們的每個防禦都比別人好」，而是「對於幾乎所有可能的輸入，我們的總損失都優於或等於任何替代方案」。

## 對實作計畫的影響

這篇研究明確了：17 個死碼模組不是「獨立的防禦工具」，而是 **6-Response 狀態機的某個節點**。整合它們時不應該「把每個模組編譯通過」，而是「以狀態機為主軸串聯它們」。

因此，v6_integration.rs 是整個架構的核心——它就是那個狀態機。它引用了幾乎所有其他模組，因為它負責將 ExecuteRequest 映射到 6 種回應之一。
