# SASS v6.0 架構矛盾與非預期成果修正研究 (Phase 9 預備)

> **建立時間**：2026-05-25 05:53 (UTC+8)
> **專案**：SakiAgentSSH
> **屬性**：Scientia (架構研究)
> **目的**：針對 v6.0 (Phase 8) 盤點中發現的三大架構矛盾，提出具體的修復方案。

---

## 矛盾一：Rust 與 Go 實作嚴重落差

**現況描述**：
目前 `go-sakissh` (Go 版 Daemon) 僅實作了基礎的 Zstd+Base64 派發，完全缺失 v5.0 的 13Policy 焦油坑，以及 v6.0 的 `BranchMgr` (微型分支)、`EnvInjector` (環境注入) 與 `AuditLogger` (密碼學稽核)。

**修正研究與方案**：
由於 Go 與 Rust 的生態系差異，完全同步實作防禦層的成本過高。建議採取以下兩種路徑之一：

- **方案 A (Go 棄用/降級)**：將 Go 版本標記為 `SASS-Lite`，僅用於資源極度受限的嵌入式設備 (如老舊的 MIPS 路由器)，並在 README 中明確聲明不具備四大再現性與沙盒防禦。
- **方案 B (CGO/FFI 共用防禦庫)**：將 Rust 的防禦邏輯 (`audit`, `branch_mgr`, `env_injector`, `policy`) 編譯為 C-compatible 的 shared library (`libsass_defense.so/dylib`)，讓 Go Daemon 透過 cgo 呼叫。這樣可以確保核心安全邏輯只有一份實作。
- **推薦決策**：**方案 A**。為了維持 Go 版本的跨平台極簡性（無需 CGO 工具鏈），應將其定位為 Lite 版本。主線防禦全押注於 Rust 版本。

## 矛盾二：BranchMgr 未落實底層 Union Mount

**現況描述**：
`branch_mgr.rs` 只是建立了一個空目錄 `/tmp/sass_branches/{session_id}` 並將其設為 `cwd`。在 macOS 上由於缺乏原生的 OverlayFS，這會導致原本在專案目錄中的檔案，在執行階段完全「消失」，引發各種 `File Not Found`。

**修正研究與方案**：

在 macOS 上要實作目錄的微型動態分支，有以下幾種技術選擇：

1. **APFS 快照與 Firmlink (蘋果原生)**：
   - 優點：底層支援，效能極高。
   - 缺點：需要特殊的 Entitlements，且操作 APFS volume 需要 root 權限，不符合 SASS 的 User-level 設計。
2. **macOS FUSE (macFUSE)**：
   - 利用 FUSE 實作一套 Userspace 的 Union File System。
   - 優缺點：需要使用者安裝 kext (macFUSE)，對自動化 Agent 不友善。
3. **Symbolic Link Tree (Symlink Overlay 演算法)**：
   - **實作原理**：遍歷目標目錄，在 `/tmp/sass_branches/{session_id}` 下為所有原目錄的檔案建立軟連結 (Symlink)。當 Agent 要修改某檔案時，SASS 攔截 (透過 `kernel_bridge` 或前置腳本) 並執行 `Copy-on-Write` (將軟連結替換為實體檔案拷貝)。
   - **優點**：純 Userspace 實作，無需 root，完全跨平台。
   - **推薦決策**：採用 **Symlink Tree (Soft-link Overlay)**。實作一個 `build_symlink_tree(src, dest)` 函數。雖然在有數十萬檔案的 `node_modules` 時建立 symlink tree 可能需要幾秒鐘，但搭配 `EnvInjector` 已經將 `node_modules` 卸載，因此實際需要 link 的原始碼檔案數量極少。

## 矛盾三：Audit Key 缺乏持久化驗證機制

**現況描述**：
`audit.rs` 每次啟動時透過 `SigningKey::generate(&mut OsRng)` 產生記憶體金鑰。私鑰無法持久化，也無法匯出公鑰，導致第三方稽核者無法驗證 `audit.jsonl` 中 Ed25519 簽章的合法性。

**修正研究與方案**：

為了讓「可完整稽核」具有意義，公鑰必須能被第三方取得，且私鑰必須安全地持久化。

1. **Keychain / Secure Enclave 整合 (macOS/Windows 專屬)**：
   - 在 macOS 上使用 `security` framework，將 Private Key 生成於 Secure Enclave 內部，SASS 僅取得 Reference 進行簽章。
   - 優點：私鑰絕對安全，連 SASS 自身也無法讀取私鑰明文。
2. **Local PEM File (跨平台通用)**：
   - 第一次啟動時，SASS 產生 Ed25519 Keypair，將 Private Key 加密 (或 chmod 400) 存入 `~/.config/sass/audit_key.pem`。
   - 將 Public Key 存入 `~/.config/sass/audit_pub.pem` 並提供 RPC 端點 (`GetAuditPublicKey`) 供 Client 查詢。
3. **推薦決策**：實作 **Local PEM File + RPC PublicKey 廣播**。
   - 修改 `config.rs` 新增 `audit_key_path`。
   - 在 `AuditLogger::new` 中，若金鑰不存在則產生並寫入檔案；若存在則讀取。
   - 為了安全，啟動時強制檢查該 `.pem` 的檔案權限，若非 0600 或 0400 則拒絕啟動。
