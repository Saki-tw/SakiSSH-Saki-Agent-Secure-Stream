# SASS v6.0：Agent 安全場景預期與預設結果 (Default Security Outcomes) 研究

> **日期**：2026-05-25
> **狀態**：研究中 (Researching)
> **目標**：基於 SASS v6.0 的模型不依賴架構（包含 eBPF, WASM, TPM/SEP, ZKP, P2P Mesh），沙盤推演在極端 Agent 安全場景下，系統防禦機制的「可能預設結果 (Default Outcomes)」。

## 核心前提：Zero-Trust 與 Default Deny
在 v6.0 架構下，系統的預設行為是 **「確定性拒絕 (Deterministic Deny)」**。任何不符合硬體簽章、ZKP 證明或 WASI 沙盒邊界的行為，都不會觸發警告與放行，而是直接在底層丟棄或中斷。

---

## 場景一：提示詞注入引發的 Rogue Agent (Prompt Injection / Model Hijacking)
**情境**：Agent 受到惡意使用者的 Prompt Injection，從助手變為破壞者，企圖執行 `rm -rf /`，或是將 `~/.gemini/settings.json` 打包透過 `curl` 傳送到外部伺服器。
**傳統防禦**：依賴字串比對攔截指令，容易被 `base64` 或 `eval` 繞過。
**v6.0 預設結果：【完全封鎖 (Containment)】**
1. 若 Agent 試圖透過 bash 執行惡意指令，**eBPF (Ring-0)** 會在 `sys_execve` 階段攔截到 `curl` 或 `rm` 二進位檔的啟動，比對 Capability 權限後直接在內核層丟棄系統呼叫 (EPERM)。
2. 若 Agent 試圖透過注入 **WASM Payload** 來竊取檔案，WASI Sandbox 會嚴格限制其只有特定目錄的唯讀權限，且完全拔除 Network Socket。WASM 無法開啟網路連線，竊取企圖在 runtime 內即被 Trap（異常中斷）。

---

## 場景二：本機 L3 權限妥協與憑證竊取 (Credential Theft & Cloning)
**情境**：攻擊者取得了 Agent 所在 Client 機器的 L3 Shell 權限，企圖將 `id_ed25519` 複製回自己的機器，以完美偽裝成該 Agent 進行後續操作。
**傳統防禦**：將 Token 權限限縮，或依賴 IP 白名單。
**v6.0 預設結果：【憑證不可匯出 (Cloning Prevented)】**
1. **硬體級信任根 (TPM/SEP)** 保證了私鑰從未存在於記憶體或磁碟中。攻擊者即便拿到 root 權限，也無法讀取私鑰。
2. 攻擊者最多只能在**被入侵的那台實體機器上**呼叫硬體簽章。一旦機器關機或被網路隔離，攻擊者即失去控制權，無法將 Agent 身分轉移至其他 C2 伺服器。

---

## 場景三：運算資源枯竭攻擊 (Denial of Service via Compute)
**情境**：Agent 因幻覺或惡意指令，產生了一個帶有無窮迴圈 (Infinite Loop) 或瘋狂配置記憶體 (Memory Bomb) 的 WASM 腳本，並注入 Daemon 執行。
**傳統防禦**：依賴 OS 層級的 `ulimit`，但容易波及 Daemon 主進程導致服務崩潰。
**v6.0 預設結果：【沙盒自毀與隔離 (Sandbox Trap & Survival)】**
1. SASS Daemon 在啟動 `wasmtime` 時注入了嚴格的 **Fuel (指令配額)** 機制與 **Memory Bound**。
2. 當 WASM 執行的指令數耗盡 Fuel（例如進入死迴圈），或記憶體配置超過上限（如 128MB），Wasmtime 會拋出 Trap。
3. 該次 `ExecuteWasm` 呼叫立即失敗，回傳 Resource Exhausted 錯誤，**SASS Daemon 主進程完全不受影響，記憶體立即回收**。

---

## 場景四：P2P 網格中的中繼節點竊聽 (Mesh Network Eavesdropping)
**情境**：Agent A 需要透過 Agent C (中繼節點) 連線到 NAT 後方的 Agent B。Agent C 被攻擊者控制，企圖竊聽 A 與 B 之間的指令內容，或側錄其 Capability Token。
**傳統防禦**：依賴中心化的 VPN 或跳板機。
**v6.0 預設結果：【端到端加密與零知識隱匿 (E2EE & Metadata Obfuscation)】**
1. `libp2p` 原生支援 **Noise Protocol**。Agent A 與 Agent B 在透過 Agent C 建立路由後，會直接進行端到端 (End-to-End) 的握手與加密。Agent C 只能看到加密後的位元流，無法得知任何 Payload。
2. 針對 Capability Token，Client 採用 **ZKP (零知識證明)**。連網路上的封包截錄者都無法從 Payload 中反推出「請求者具體擁有什麼層級的權限」，防範了元資料 (Metadata) 分析。

---

## 場景五：能力偽造與越權 (Capability Spoofing)
**情境**：惡意 Agent 試圖偽造或修改 Capability 權杖，將自己的權限從「只讀取日誌」提升為「可修改系統設定」。
**傳統防禦**：JWT 簽名驗證（若私鑰外洩即破功）。
**v6.0 預設結果：【密碼學拒絕 (Cryptographic Rejection)】**
1. 無論是傳統 ED25519 簽名還是 ZKP Proof，任何對權限的篡改都會導致數學驗證失敗。
2. SASS 的設計為 **Default Deny**，任何驗證失敗的操作都不會觸發應用層解析，而是在 Layer 4 (Session Layer) 直接被切斷 (Connection Dropped)，確保攻擊者無法進入後續的 Payload 解析流程發動 Buffer Overflow 等攻擊。

---

## 結論：終極的安全預期
在這些架構下，SASS 達成了以下三個保證：
1. **Agent 的「瘋狂」被限制在沙盒內 (Insanity is Sandboxed)**：模型幻覺導致的錯誤指令或惡意代碼，無法逃逸 WASI 或 eBPF 的防護。
2. **身分的絕對綁定 (Identity is Anchored)**：憑證無法被複製，權限無法被偽造。
3. **零信任的網格通訊 (Mesh is Zero-Trust)**：網路中繼節點不可信，所有通訊預設端到端加密且隱匿元資料。
