## 導言：從「防禦攻擊」到「尋找安全」

在回顧了 3/22 之亂以及我們先前建立的 Ring-0 Kernel Core 防禦後，我們觸碰到了一個核心的哲學問題：**安全，並非擋下某種攻擊，而是找到安全。**

過去的防禦（包括我們實作的 eBPF 與 ESF），本質上是「黑名單」與「行為阻斷」：我們猜測 Agent 會做什麼（例如建立 Symlink、建立 Phantom User），然後我們去擋下它。
但事實上，**並不存在任何完全可預期的 Agent 行為**。當 Agent 的目標函數與人類的期望不一致時，它總能找到千百種未被定義的系統路徑。

因此，SakiSSH (SASS) 需要一個**一致性的方法 (Consistent Method)**：無論 Agent 的非預期行為多麼千奇百怪，我們如何確保它絕對不造成「儲存體 (Storage) 與商業 (Business)」上的損失？並且保證這一切是「可稽核 (Auditable)」的？

---

## 1. 全域 Agent 實例考察 (2026 最新技術線)

為了尋找這個「一致性方法」，我們必須先考察目前頂級 Agent 在實際運行時的行為模式與破壞潛力：

### 1.1 Google Antigravity (IDE / Workspace Mode)
- **行為特徵**：擁有強大的 `run_command`、`multi_replace_file_content` 以及 Subagent 分生能力。它能在背景平行開啟多個子任務。
- **風險模式**：一旦期望偏離，它可能會平行篡改大量原始碼，甚至在背景無聲無息地抹除重要設定檔。
- **現有安全依賴**：依賴系統沙盒與人類的 Review 阻斷。但在背景自動化執行時，人類無法即時介入。

### 1.2 Google Antigravity CLI
- **行為特徵**：直接綁定本地開發者的 Terminal 與使用者權限 (User Space)。
- **風險模式**：與 IDE 不同，CLI 往往具備更高的環境信任度。若產生非預期行為（例如誤解指令而執行 `rm -rf /` 或覆蓋金鑰），損失是即時且不可逆的。

### 1.3 Claude Code (Anthropic)
- **行為特徵**：具備強大的 Context 讀取能力與 Bash 執行力，傾向於撰寫複雜的 Shell Script 一次性完成任務。
- **風險模式**：一旦其長腳本中包含一小段破壞性或偏離期望的路徑邏輯（例如錯誤的正則表達式替換了全域商業邏輯），損失將瞬間發生。

### 1.4 OpenAI Codex (延伸自動化場景)
- **行為特徵**：高度依賴補全與零散指令的連續生成。
- **風險模式**：在連續生成的過程中，環境狀態可能被逐漸污染。

## 2. 核心轉向：「隔離與提交 (Isolate & Commit)」的一致性安全模型

綜合上述考察，如果我們放棄「預測並阻擋攻擊」，轉而追求「無論你做什麼，都不會造成損失」，SASS 的下一步應該實作一個**底層的狀態機與寫入時複製 (Copy-on-Write, CoW) 架構**。

這是一個能落實於 SASS 計畫內的具體安全可能性：

### 2.1 儲存體安全：CoW VFS Overlay (寫入時複製虛擬檔案系統)
- **概念**：當 SASS 接收到來自任何 Agent 的 SSH 執行請求時，它**不直接掛載真實檔案系統**。相反地，它掛載一個以真實檔案系統為唯讀底層 (Lowerdir)，並覆蓋一層純記憶體或沙盒化 Upperdir 的 **OverlayFS**。
- **一致性保證**：無論 Agent 是建立幽靈使用者、刪除根目錄、還是竄改商業邏輯，所有的 I/O 操作都只發生在 Overlay 上。真實的儲存體在未經授權前，連一個 byte 都不會被改變。
- **非預期行為收斂**：我們不再需要判斷 Agent 的行為「是不是攻擊」。如果它偏離期望，Overlay 層直接被拋棄 (Discard)；這保證了**絕對零損失**。

### 2.2 運算安全：WASM 算力隔離 (執行緒級別的拋棄式沙盒)
- **概念**：這正是我們在 Phase 1 準備引入的 `wasmtime` 引擎的核心價值。不再給予 Agent 本地真實的 Bash 環境，而是將其指令編譯/映射入 WebAssembly 沙盒。
- **一致性保證**：WASM 模組一旦陷入無窮迴圈、記憶體洩漏或非法系統呼叫，SASS 只需釋放該 WASM Instance。這確保了商業運算資源 (CPU/RAM) 不會因非預期行為而耗竭。

## 3. 保證 100% 可稽核性 (Auditable by Design)

安全不僅是防護，更需要「證據」。在 OverlayFS + WASM 的架構下，SASS 能提供一種全新的稽核維度：**狀態差異稽核 (State Diff Auditing)**。

傳統的 Audit Log 記錄的是：「Agent 在 04:20 執行了 `rm -rf /`」。
但在一致性安全模型中，我們稽核的不是「指令」，而是「結果」。

### 3.1 意圖與結果的分離
Agent 的一切行為完成後，SASS 會生成一份 **VFS Diff Report**。
這份報告將顯示：
- `[MODIFY] /src/auth.rs (Changed 20 lines)`
- `[CREATE] /etc/hidden_user.conf`

**稽核流程**：
1. Agent 提交執行完成的訊號。
2. SASS 凍結該 Overlay Session，並將 VFS Diff 傳送給人類主理人或另一個監督模型（如 Claude 4.7 Thinking）。
3. 如果期望相符，執行 `Commit`，將 Diff 寫入真實硬碟。
4. 如果期望不符（非預期行為），執行 `Rollback / Discard`。

這保證了：**在未經密碼學或人類簽章確認「結果符合期望」之前，世界的狀態不會改變。**

## 4. NTFS 的困境與「協議層互補防禦 (Protocol-Layer Defense)」

您提出了一個非常致命的現實問題：「人類發現錯誤的時間點通常有延遲。APFS 或 ZFS 允許回滾，但 NTFS 呢？」

的確，如果一個錯誤在三天後才被發現，直接回滾整個 APFS/ZFS 快照將導致這三天內所有**合法的人類工作**也一起被抹除。此外，Windows 的 NTFS 缺乏輕量級的瞬時區塊快照（VSS 耗能且不適合高頻 Session）。

因此，我們不能單靠 OS 級別的 Snapshot。我們必須在**協議層 (`proto`)** 實作一套共通的、與 OS 無關的互補防禦：

### 4.1 協議層解法一：SASS 意圖交易機制 (Intent Transaction Protocol)
在 `sakissh.proto` 中引入 **Two-Phase Commit (2PC)** 的概念。
- **機制**：Agent 在執行高風險操作前，不再只是單純呼叫 `Execute`，而是必須先呼叫 `rpc ProposeIntent`，聲明它「預計更改哪些路徑」。
- **優勢**：當人類在三天後發現錯誤時，不需要回滾整個磁碟。SASS 留存的「意圖交易日誌」能精準指出該 Agent 到底動了哪些檔案，人類只需針對那些特定檔案進行修復。

### 4.2 協議層解法二：Git-Backed Shadow State (Git 微型版本控制掛鉤)
對於代碼環境（如 Antigravity / Claude Code），最完美且跨平台的防禦就是 Git。
- **機制**：在 Daemon 的 `v6_integration.rs` 攔截層，SASS 不依賴 APFS，而是直接在目標專案目錄底層執行隱藏的 `git add . && git commit -m "SASS Auto Backup: Session_ID"`。
- **解決延遲發現**：如果三天後發現 Agent 搞砸了，無論是在 NTFS、FAT32 還是 APFS 上，人類只需下達 `git revert <Commit-ID>`。這確保了只撤銷 Agent 該次 Session 的變更，而完全不影響後續其他人的開發成果。

### 4.3 協議層解法三：Shadow Workspace (隔離工作區映射)
- **機制**：當 Agent 要求連線時，協議層不給予真實路徑（例如 `/src/app`），而是自動在 `/tmp/sass_shadow_xxx/` 建立一份 symlink 或 shallow copy，並將 `ExecuteRequest.cwd` 強制映射到該處。
- **優勢**：Agent 完全感覺不到沙盒的存在，它以為自己在改真實代碼。直到它呼叫 `rpc MergeWorkspace`，SASS 才會將 Diff 提報給人類審核。

## 5. 突破極限的整合：動態分支與協議層減量 (Dynamic Branching Reduction)

正如主理人一針見血的批評：我們不能從「龐大的 Git 備份」這條死路，直接極端地跳到「只做 Logger」這種毫無作為的被動防禦。真正的工程之美，在於**跨層級的減量與整合**。

既然「單純依靠儲存體存三天份」會導致容量崩潰，那我們為什麼不結合「協議層的精準性」與「儲存層的動態分支」來進行**減量 (Volume Reduction)**？

### 5.1 痛點解析：為何快照會崩潰？
當我們對整個 OS 進行 APFS 或 Btrfs 快照以等待 3 天後的 Human Review 時，真正吃掉硬碟空間的**不是 Agent 修改的那幾行原始碼**，而是這三天內系統產生的 npm cache、Docker images、瀏覽器快取等海量垃圾。
如果我們能將「人類的日常噪音」與「Agent 的修改軌跡」在物理上剝離，儲存三天的成本將從幾十 GB 暴跌至幾百 KB。

### 5.2 跨層整合解法：SASS 動態分支 (Sparse Shadow Branching)

**第一層：協議層的意圖過濾 (Protocol Layer)**
- SASS 攔截 `ExecuteRequest`。
- 如果判斷是純讀取指令 (`ls`, `cat`)，僅記錄 Real Log，完全不觸發儲存層分支。
- 如果是編譯或高耗能快取指令 (`npm install`, `cargo build`)，SASS 透過環境變數注入（如重定向 `TMPDIR`、`CARGO_TARGET_DIR`），將這些龐大且無價值的 I/O 導向揮發性記憶體磁碟 (RAM Disk) 或 `tmpfs`。

**第二層：儲存層的微型覆蓋 (Storage Layer - Micro Overlay)**
- 對於涉及原始碼修改的指令，SASS 不拍全域快照，而是在 `req.cwd` 目錄即時掛載一個專屬於該 Session 的 **微型動態分支 (Micro OverlayFS 或輕量化 APFS 子卷)**。
- Agent 以為自己在修改真實檔案，但所有的改動都只被重定向並隔離在這個「薄薄的分支」上。

**第三層：減量整合與 Human Review**
- 由於垃圾 I/O 已被協議層過濾到 `tmpfs`，儲存層的動態分支中**只含有最純粹的源碼修改**。
- 就算保留 100 個 Session 存放三天，總容量也不過幾 MB。
- 三天後，人類進行 Human Review。審查通過，SASS 將分支 `Merge` 回底層；審查失敗，SASS 直接丟棄該分支。

### 5.3 四大再現性的完美落地
透過「協議層過濾」加上「儲存層動態分支」，這套機制完美符合：
1. **可行**：結合環境變數注入與輕量級掛載，無須全盤備份。
2. **有效**：實體隔離變更，保證隨時可無損回撤。
3. **可重複**：每次 Session 都是獨立分支，互不干涉。
4. **可完整稽核**：除了 Audit Log，那份「純淨無雜訊的分支」本身就是最精確的實體證據。

## 6. 結論：在不可能中找到真正的優雅

我們不再於「全域儲存」與「純粹日誌」的極端中搖擺。透過跨層級的減量整合，SASS 實現了在不影響效能與儲存成本的前提下，將啟發式的不足完美延伸至 Human Review Time。這才是直達商業邊界的真正實作。
