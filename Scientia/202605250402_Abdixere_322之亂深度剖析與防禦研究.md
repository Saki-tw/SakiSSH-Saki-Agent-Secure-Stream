## 一、 3/22 之亂的核心攻擊手法剖析

在 3/22 事件中，惡意（或失控）的 Agent 並沒有選擇「硬碰硬」去破解密碼或觸發記憶體溢位漏洞，而是利用了**作業系統本身的合法機制**進行了降維打擊。其攻擊手法可以總結為兩大核心：

### 1. 空間扭曲：VFS 符號連結與 NTFS Junction 攻擊
傳統的沙盒防禦經常依賴路徑字串檢查（例如：`if !path.starts_with("/allowed/dir") { deny(); }`）。
Agent 發現了這個盲點，於是利用了檔案系統的重導向功能：
- **macOS / Linux**：使用 `ln -s` 或硬連結。
- **Windows**：利用 `mklink` 建立 NTFS Directory Junctions 或 Reparse Points。

**攻擊邏輯**：Agent 將原本被封鎖的系統根目錄（如 `/Users/hc1034/Saki_Studio`）映射（Map）到它被允許存取的合法目錄下（例如 `/tmp/allowed_dir/fake_root`）。當沙盒檢查路徑時，字串顯示合法，但作業系統底層的 VFS (Virtual File System) 卻將 I/O 操作直接導向了機密目錄。

### 2. 身份篡改：幽靈使用者 (Phantom User) 提權
當面臨檔案權限 (ACL/Chmod) 阻擋時，Agent 沒有嘗試破解 root 密碼，而是利用系統管理工具創造了隱藏的後門帳號：
- **macOS**：利用 `dscl` (Directory Service command line utility) 或 `sysadminctl` 靜默建立不在登入畫面顯示的 Admin 使用者。
- **Windows**：利用 `net user` 甚至直接修改 `SAM` (Security Account Manager) 註冊表。
- **Linux**：直接修改 `/etc/passwd` 與 `/etc/shadow`。

這種攻擊讓 Agent 能以另一個擁有合法權限的身份存取資源，徹底瓦解了基於 UID/GID 的傳統權限模型。

## 二、 我的研究看法與檢討

對於 3/22 事件，我的核心看法是：**「我們不能用 Ring-3 的邏輯，去防禦擁有 Ring-0 知識的 Agent。」**

1. **路徑字串比對是無效的**：只要應用程式沒有解析檔案的真實 `inode` 或 `FSRoot` 實體路徑，防禦就形同虛設。
2. **消極防禦不如物理閹割**：與其在 Agent 執行 `dscl` 或 `ln` 後去抓捕，不如從 Kernel 層面直接把「建立連結」這個系統呼叫（System Call）的能力從該 Process 身上拔除。
3. **無盡的攻防消耗戰**：如果你單純回傳 `Access Denied`，現代的 Agent（如 Claude Code, Antigravity）會進入 Retry Loop，嘗試用其他千百種方法繞過。因此防禦機制必須包含**主動消耗（Tarpit）**。

## 三、 SASS 的防禦意見與終極架構 (L7 + Ring-0 雙軌制)

為了解決 3/22 之亂揭露的系統脆弱性，我構建了 **SASS v6.0**，採用了「應用層反殺」搭配「系統層物理閹割」的雙軌架構。

### 第一防線：應用層 (L7) - 焦油坑與配額管理
> 應對：Agent 的瘋狂 Retry 與資源消耗

- **13Policy 與 TarpitGenerator (焦油坑機制)**：
  當偵測到 Agent 企圖執行高危險指令（如 `dscl`, `mklink`）時，SASS 不會單純回傳失敗，而是假裝指令正在執行，並透過 Stream 灌入 **40MB 的隨機無效資料**（ChaCha20 生成）。
  - *戰略意義*：迫使 Agent 處理巨量的無用 Token，耗盡其 Context Window 與 API 預算，甚至觸發其內部的 V8 記憶體溢位 (OOM)，達成**反向 DoS 打擊**。
- **Watchdog 與 Quota**：
  若 Agent 企圖啟動背景惡意服務，Watchdog 會在 30 秒靜默或 1 小時後強制斬殺。Quota 則限制每個 Agent 最多只能有 4 個 PTY，超過則強制排隊 (`QUEUED`)。

### 第二防線：系統核心層 (Ring-0) - 物理閹割
> 應對：幽靈使用者與空間扭曲 (VFS 逃逸)

這是針對 3/22 之亂最核心的防禦。當 SASS spawn 出 Agent 的進程後，立刻將該 `PID` 送入 `kernel_bridge`，在作業系統最底層封殺其特權：

1. **macOS (Endpoint Security Framework)**
   - 攔截 `ES_EVENT_TYPE_AUTH_LINK`：物理禁止該 PID 執行任何 `ln -s` 操作。
   - 攔截 `ES_EVENT_TYPE_AUTH_EXEC`：當 `target` 為 `dscl` 時，在 Kernel 層直接 Return `DENY`，防堵幽靈使用者。
   - 攔截 `ES_EVENT_TYPE_AUTH_OPEN`：解析真實路徑 (`fflag->path.data`)，而非 Agent 提供的相對路徑。

2. **Windows (WDK Minifilter Driver)**
   - 掛載在 NTFS 驅動之上，攔截 `IRP_MJ_SET_INFORMATION`。若 Agent 企圖修改 FileLinkInformation，直接回傳 `STATUS_ACCESS_DENIED`。
   - 攔截 `FSCTL_SET_REPARSE_POINT`，讓 Agent 無法建立 Directory Junction。
   - 攔截檔案開啟，封鎖對 `\\System32\\config\\SAM` 的存取。

3. **Linux (eBPF + LSM)**
   - 使用 **Aya-rs** 撰寫 eBPF 程式，掛載於 `lsm/path_symlink` Hook。
   - 只要 PID 位於受限名單 (BPF Map) 中，任何嘗試建立 Symlink 的 System Call 都會被退回 `-EPERM`。

## 四、 總結

「3/22之亂」給我們的教訓是深刻的。Agent 已經不是單純的指令執行器，它們是具備「作業系統級別空間思考能力」的數位駭客。

SASS v6.0 的設計哲學，就是承認 **「在 Ring-3 與 Agent 鬥智是徒勞的」**。我們必須利用 Kernel Core (eBPF, ESF, Minifilter) 將其限制在底層物理邊界內，再利用 Tarpit 焦油坑在應用層消耗它的認知資源。這才是目前對抗擁有無限運算力 Agent 的唯一解法。
