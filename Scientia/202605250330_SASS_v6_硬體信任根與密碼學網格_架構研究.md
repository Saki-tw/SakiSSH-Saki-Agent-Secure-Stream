# SASS v6.0：硬體信任根、零知識證明與 P2P 網格架構研究

> **日期**：2026-05-25
> **狀態**：研究中 (Researching)
> **目標**：提升 SASS 的身分認證極限（硬體綁定與隱私保護），並打破 Client-Daemon 的拓樸限制，建立去中心化的 Agent 溝通網格。

## 1. 硬體級信任根 (Hardware Root of Trust)

### 1.1 背景與痛點
SASS v5.0 的認證基於 `id_ed25519`。只要該檔案存在於磁碟上，就有被 L3 級別漏洞（例如權限提升、目錄穿越）竊取的風險。一旦憑證被竊，攻擊者就能偽裝成合法 Agent。

### 1.2 跨平台硬體支援方案

| 作業系統 | 安全晶片 | 整合方案 (Rust 生態系) | 實作難度與挑戰 |
|----------|----------|------------------------|----------------|
| **Windows** | TPM 2.0 | `tss-esapi` crate (TCG 官方 API 封裝) | 高。需要處理複雜的 TPM Context、NVRAM 讀寫與授權 Session。 |
| **macOS** | Secure Enclave (SEP) | `security-framework` crate (封裝 `SecKeyGeneratePair` 等 API) | 中。蘋果的 API 較為高階，但只支援 P-256 (Prime256v1) 橢圓曲線，不支援 ED25519。 |
| **Linux** | TPM 2.0 | 同 Windows (`tss-esapi`) | 高。需確保 `tpm2-abrmd` 資源管理器可用。 |

### 1.3 架構適配與變更
因 Secure Enclave 不支援 ED25519，SASS 的 Auth 流程必須從單一 ED25519 擴充為支援多種簽章演算法 (Agile Cryptography)：
- 允許 `AuthRequest` 夾帶 ECDSA P-256 簽章。
- Daemon 端在註冊 Agent 時，需記錄該 PublicKey 對應的硬體型態。
- 這樣能確保憑證「不可匯出 (Non-exportable)」，甚至連管理員也無法複製憑證。

## 2. 零知識證明 (ZKP) 權限驗證

### 2.1 背景與優勢
在高度敏感的環境（例如 SakiMed 醫療系統），不僅操作內容需要保密，連「哪個 Agent 持有什麼權限」的元資料 (Metadata) 也不應在網路上明文傳輸。
透過 ZK-SNARKs，Agent 可以證明「我知道一個合法的私鑰，且該私鑰被記錄在 Daemon 的 Merkle Tree 中」，而完全不暴露私鑰對應的公鑰或 ID。

### 2.2 技術線：Halo2 / Bellman

- **Halo2**：Zcash 開發的無信任設定 (Trusted Setup) 證明系統。
- **Bellman**：較早期的 zk-SNARK 庫，基於 Groth16。

**架構設計**：
1. **Setup**：管理員產生一個 Merkle Tree，葉子節點是所有合法 Agent 的公鑰 Hash。樹根 (Root) 被存放在 Daemon 中。
2. **Proof Generation (Client 端)**：Agent 使用 ZK 電路，輸入私鑰、公鑰與 Merkle Path（皆為 Private Input），產生一個 Proof。
3. **Verification (Daemon 端)**：Daemon 收到 Proof 與 Merkle Root（Public Input），驗證其有效性。
4. **結果**：Daemon 知道「某個合法的 Agent」請求了權限，但不知道具體是誰。

### 2.3 效能衝擊評估
- Proof 生成在 Client 端（可能是 M1 晶片），預計耗時數秒。對於非高頻連線可接受。
- Proof 驗證在 Daemon 端，僅需數毫秒。
- 缺點：依賴庫極度肥大，將大幅增加 `saki-ssh-client` 與 `daemon` 的編譯時間與 Binary 體積。

## 3. 去中心化智能網格 (SASS P2P Mesh Network)

### 3.1 背景與痛點
目前的 Client-Daemon 拓樸在跨網域時（例如公網 Agent 欲控制內網 U9 機器），需要依賴 SSH 隧道或中繼伺服器，無法自動化穿透 NAT。

### 3.2 網格架構設計 (基於 libp2p)

引入 `rust-libp2p`，將 SASS 從 C/S 架構轉變為 P2P 架構：

- **節點身分 (PeerId)**：直接由現有的 ED25519 公鑰推導而來。
- **發現協定 (Discovery)**：
  - 區域網路：使用 mDNS 自動發現同網段的 Daemon。
  - 廣域網路：使用 Kademlia DHT，Agent 啟動時連接已知的 Bootstrap 節點。
- **穿透與路由 (Hole Punching & Relay)**：
  - 實作 DCUtR (Direct Connection Upgrade through Relay) 協定。
  - 當 Agent A 無法直接連線 Agent B，可透過公網的 Agent C 作為 Relay。

### 3.3 RPC 演進 (gRPC over libp2p)
- 原有的 gRPC 是架構在 TCP/TLS 1.3 之上。
- `libp2p` 原生支援多路復用 (Yamux/Mplex) 與安全傳輸 (Noise Protocol)。
- 技術挑戰：需實作一個客製化的 `tonic` Transport，將 gRPC 的 HTTP/2 流量封裝進入 libp2p 的 Stream 中。這在 Rust 中是可行的，但需要深度的非同步流 (Async Stream) 處理技巧。

## 4. 總結
這三個方向均屬於「基礎設施革命」：
- **硬體信任根** 能徹底封殺憑證外洩。
- **ZKP** 將 SASS 推向金融/醫療級別的零信任 (Zero-Trust) 極致。
- **P2P Mesh** 則能解決 Saki Studio 跨機協作與分散式運算的最後一哩路。
建議初期先從 **P2P Mesh** 的 `libp2p` 整合開始，因為它能直接解決跨 NAT 部署的痛點，最符合「啟發式開發」的實用精神。
